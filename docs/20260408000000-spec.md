# BigQuery SQL Test Runner 仕様書

## 1. 目的

BigQuery 上で動作する既存の SQL を対象に、以下を可能にする。

- 本番テーブル参照をテスト用入力に差し替える
- BigQuery の実エンジンでクエリを実行する
- 実行結果を期待値と比較する
- 差分を見やすく表示する
- CI 上で成功/失敗判定に使える

> このツールは BigQuery のエミュレーションを目指すものではなく、BigQuery 本体を使ったクエリテストのハーネスである。

## 2. 想定ユースケース

| ユースケース | 説明 |
|---|---|
| 単体的な SQL テスト | 複雑な SELECT クエリに対して小さい fixture を与えて期待結果を検証 |
| 回帰テスト | 既存 SQL のロジック変更時に以前通っていたケースが壊れていないことを確認 |
| 境界ケース検証 | NULL、重複、欠損、0件、JOIN 不成立、日付境界などを fixture で明示的に与えて確認 |
| リファクタ安全性確保 | SQL の最適化や構造変更後も意味が変わっていないことを保証 |

## 3. スコープ

### 3.1 対象

- BigQuery の GoogleSQL
- 主に SELECT 系クエリ
- 必要に応じて `INSERT INTO ... SELECT`, `CREATE TABLE AS SELECT` の読み取り元解析も対象

### 3.2 非対象

初期バージョンでは以下は明示的に非対応または限定対応とする。

- dynamic SQL
- `EXECUTE IMMEDIATE`
- スクリプト中で文字列として構築される SQL
- 外部テーブル
- 本番書き込みを伴う DML/DDL の完全自動テスト
- 権限のない cross-project 参照の完全透過処理
- BigQuery 新構文への即時フル追随保証

## 4. 基本コンセプト

処理フロー:

1. SQL を BigQuery 方言として解析する
2. SQL 内の実テーブル参照を抽出する
3. 指定された fixture 定義に基づいて参照先を書き換える
4. BigQuery script を生成する
   - script 内で fixture 用 TEMP TABLE を作る
   - 書き換え後 SQL を実行して actual を得る
   - expected と比較する
5. 成功/失敗を返す

## 5. 用語定義

| 用語 | 定義 |
|---|---|
| source table | `FROM`, `JOIN` 等で読み込まれる実テーブル |
| destination table | `INSERT INTO`, `CREATE TABLE`, `MERGE INTO` などで書き込み先になるテーブル |
| fixture | テスト用入力データ。通常は source table に対応する小規模データ |
| expected | 期待される出力結果 |
| actual | テスト対象 SQL を fixture 上で実行した結果 |
| rewrite | 本番テーブル参照を fixture 用の TEMP TABLE 名等へ置換すること |

## 6. アーキテクチャ概要

### 6.1 SQL Parser / Analyzer

- SQL を AST に変換する
- 実テーブル参照を列挙する
- CTE 名や alias と実テーブルを区別する
- source / destination の文脈を識別する

### 6.2 Rewrite Engine

- テーブル参照を fixture 用名に置換する
- 必要なら dataset 単位置換も行う
- 一部テーブルだけ差し替え、他はそのまま、を可能にする

### 6.3 Fixture Builder

- fixture 定義から `CREATE TEMP TABLE ... AS SELECT * FROM UNNEST(...)` 等を生成する
- スキーマ不一致があれば検出する
- JSON / YAML から生成する

### 6.4 Expected Builder

- expected 定義を TEMP TABLE 化する
- 行順を無視する比較を基本とする
- 必要なら順序付き比較も提供する

### 6.5 Comparator

- `actual EXCEPT DISTINCT expected`
- `expected EXCEPT DISTINCT actual`
- をベースに差分を算出する

### 6.6 Executor

- BigQuery API 経由で script を実行する
- job status を管理する
- エラー時の SQL と失敗箇所を出す

### 6.7 Reporter

- 成功/失敗
- 差分件数
- 差分行
- 使われた fixture
- 書き換え後 SQL
- などを表示する

## 7. テストモデル

1 テストケースは最低限次の要素を持つ。

- テスト名
- 対象 SQL
- fixture 一式
- expected
- 比較条件
- 実行オプション

## 8. テストケース仕様

### 8.1 metadata

- `test_name`
- `description`
- `tags`

### 8.2 target

- sql file path または raw SQL
- `dialect = bigquery`

### 8.3 fixtures

テーブルごとの入力データ。

- fully qualified table name
- temp table name
- rows
- 任意で schema

例: `myproj.raw.orders` → `orders`, `myproj.raw.users` → `users`

### 8.4 expected

- rows
- 任意で schema
- 任意で sort key

### 8.5 compare options

- unordered set compare
- ordered compare
- nullable strictness
- float tolerance
- timestamp tolerance

### 8.6 execution options

- location
- project
- labels
- dry run
- keep generated script

## 9. fixture 定義方式

### 9.1 第一候補: YAML/JSON の行定義

```yaml
fixtures:
  - table: myproj.raw.orders
    temp_name: orders
    rows:
      - {order_id: 1, user_id: 10, amount: 100}
      - {order_id: 2, user_id: 10, amount: 200}
```

- 利点: 宣言的、Git 管理しやすい、差分が見やすい
- 欠点: 型表現がやや面倒、STRUCT/ARRAY は少し冗長

### 9.2 SQL fixture

fixture 自体を SQL で書く。

- 利点: BigQuery 型との相性がよい、ARRAY/STRUCT に強い
- 欠点: 長くなりやすい

> 初期仕様では YAML/JSON + 必要に応じて SQL fixture を推奨とする。

## 10. テーブル参照の抽出仕様

### 10.1 方式

- 文字列置換ではなく **AST ベース**
- BigQuery 方言として parse
- Table 相当ノードを列挙する

### 10.2 抽出対象

- `FROM`
- `JOIN`
- サブクエリ内の `FROM`
- CTE 内部の `FROM`
- 必要に応じて `MERGE USING`

### 10.3 抽出対象外

- CTE 名
- subquery alias
- 列名
- `UNNEST(...)`
- temp table 名
- function 名

### 10.4 source / destination の区別

- 書き込み先は source table とは別扱いにする
- 初期版では source のみ rewrite 対象とする

## 11. rewrite 仕様

### 11.1 基本

fixture が定義されている source table だけを rewrite する。

例: `myproj.raw.orders` → `orders`, `myproj.raw.users` → `users`

### 11.2 rewrite 単位

- fully-qualified table 単位
- 将来的には dataset 単位置換も可能

### 11.3 rewrite 対象外

- fixture が定義されていないテーブル
- 明示的に passthrough 指定されたテーブル

### 11.4 失敗条件

- SQL 内に参照されるテーブルがあるのに fixture 未定義で、かつ passthrough 禁止設定
- 同一 temp table 名の衝突
- rewrite 後に識別子競合が生じるケース

## 12. 比較仕様

### 12.1 デフォルト比較

- 行順は無視
- 列名と値の一致を見る
- 差分 0 件で成功

### 12.2 ordered 比較

必要なら sort key を指定して順序込み比較する。

### 12.3 浮動小数比較

将来的に tolerance を持てるようにする（absolute / relative tolerance）。

### 12.4 NULL 比較

BigQuery の結果整形に従って通常の等値比較を行う。

### 12.5 差分表示

最低限以下を表示する:

- actual にしかない行
- expected にしかない行
- diff count

## 13. 実行生成物仕様

ツールは内部的に BigQuery script を組み立てる。構成:

1. TEMP TABLE 作成群（fixture）
2. expected TEMP TABLE 作成
3. actual TEMP TABLE 作成（書き換え後 SQL を実行）
4. 差分計算
5. ASSERT または結果返却

### 13.1 実行モード

| モード | 説明 |
|---|---|
| report | 差分を返す |
| assert | 不一致なら job fail |
| debug | 書き換え後 SQL を表示 |

## 14. CLI 仕様

### 14.1 基本コマンド

```
bqtest run <testfile>       # テスト実行
bqtest list <path>          # テストケース一覧
bqtest inspect <testfile>   # テストケースの解析結果を表示
bqtest rewrite <sqlfile>    # 書き換え後SQLを表示
```

### 14.2 主なオプション

| オプション | 説明 |
|---|---|
| `--project` | BigQuery プロジェクト指定 |
| `--location` | BigQuery ロケーション指定 |
| `--update-golden` | expected を実行結果で更新 |
| `--keep-script` | 生成された BigQuery script を保存 |
| `--debug` | デバッグ情報を表示 |
| `--fail-fast` | 最初の失敗で停止 |

### 14.3 終了コード

| コード | 意味 |
|---|---|
| 0 | success |
| 1 | test failure |
| 2 | configuration / parsing error |
| 3 | execution error |

## 15. Python API 仕様

CLI だけでなく Python からも呼べるようにする。

### 主な責務

- SQL 解析
- rewrite
- fixture から script 生成
- BigQuery 実行
- diff 結果返却

### 戻り値

- success
- diff summary
- rewritten sql
- referenced tables
- job id
- generated script

## 16. エラー設計

| エラー | 説明 |
|---|---|
| Parse Error | SQL を BigQuery 方言として解釈できない |
| Unknown Table Error | fixture も passthrough 設定もない参照テーブルが見つかった |
| Fixture Schema Error | fixture の列構成や型が不正 |
| Rewrite Conflict Error | 複数テーブルが同じ temp 名に rewrite される |
| Execution Error | BigQuery 実行時エラー |
| Assertion Failure | actual と expected が一致しない |

- actual preview

- diff preview

## 18. セキュリティ/安全性

### 18.1 本番汚染防止

- 原則 TEMP TABLE のみ使用
- destination table を自動で本番に書かない
- DML/DDL は初期版では禁止または明示許可制

### 18.2 実行先制御

- project / location を明示的に指定
- デフォルトで安全な test project を推奨

### 18.3 passthrough 制限

- fixture 未定義テーブルを本番参照してしまう事故を防ぐため、**デフォルトは禁止**
- 必要時のみ明示許可

## 19. 制約・前提

- BigQuery の実行権限が必要
- fixture が小さいことを前提とする
- テスト対象 SQL は基本的に deterministic であることが望ましい
- `CURRENT_TIMESTAMP()` など非決定性関数は固定注入またはラップを推奨

## 20. 非決定性への対応

### 対象

- `CURRENT_DATE()`
- `CURRENT_TIMESTAMP()`
- `RAND()`
- `GENERATE_UUID()`

### 対応方針

- 初期版では「テスト対象 SQL で直接使わない」規約
- 将来的には context injection を検討（例: `run_date` を外から与える）

## 21. 推奨運用規約

1. **fully-qualified name を使う** - 参照テーブルは曖昧にしない
2. **dynamic SQL を避ける** - 解析可能性を上げる
3. **source table と destination table を明確にする** - rewrite の安全性が上がる
4. **非決定関数を直接埋め込まない** - テストの再現性を保つ
5. **テスト対象ロジックは SELECT として分離** - 書き込み部分とロジック部分を分離するとテストしやすい

## 22. MVP 範囲

### MVP に入れるもの

- [x] raw SQL を入力
- [x] BigQuery 方言で parse
- [x] source table 抽出
- [x] fixture 定義されたテーブルのみ TEMP TABLE に rewrite
- [x] expected との unordered compare
- [x] CLI で 1 テスト実行
- [x] 差分表示
- [x] CI 用終了コード

### MVP ではやらないもの

- dynamic SQL
- ordered compare
- float tolerance
- golden file 自動更新
- destination table の安全な仮想化
- 列 lineage
- query plan 検査

## 23. 将来拡張

| 機能 | 説明 |
|---|---|
| snapshot / golden 管理 | expected を自動生成して更新する機能 |
| dataset clone モード | TEMP TABLE ではなく test dataset に materialize するモード |
| lineage 表示 | 「この SQL はどの source table に依存しているか」を表示 |
| lint / guard 機能 | 本番参照禁止、`SELECT *` 禁止、非決定関数警告、cross-project 参照警告 |

## 24. まとめ

> このツールは、「BigQuery SQL を AST ベースで解析し、参照テーブルをテスト fixture に差し替え、BigQuery 本体で実行し、expected と diff 比較するテストランナー」である。
